# app/ の Python を 3.11 から 3.14 へアップグレードする（PR #277 のビルド修正）

## 背景・課題

dependabot（docker ecosystem）が `app/Dockerfile` のベースイメージを
`python:3.11.11-slim-bookworm` → `python:3.14.6-slim-bookworm` へ上げる PR #277 を
作成したが、Docker image CI が失敗していた。

```text
Using CPython 3.11.15
error: Project virtual environment directory `/usr/local` cannot be used because
it is not a compatible environment but cannot be recreated because it is not a
virtual environment
```

## 原因

`app/Dockerfile` は `UV_PROJECT_ENVIRONMENT=/usr/local` により、venv を作らず
システム Python へ直接 `uv sync` する設計（2026-07-11_uv-lockfile-dependabot.md 参照）。
この設計は **ベースイメージの Python と `requires-python` の厳密一致** が前提になる。

PR #277 は Dockerfile のみ更新し、`app/pyproject.toml` の
`requires-python = "==3.11.*"` と uv.lock が 3.11 のままだったため、

1. /usr/local の Python 3.14.6 が `==3.11.*` と非互換と判定される
2. uv が CPython 3.11.15 をダウンロードし、/usr/local を venv として作り直そうとする
3. /usr/local は venv ではないため上記エラーで失敗する

つまり dependabot の docker ecosystem はベースイメージだけを上げるので、
**Python のメジャーバンプは pyproject.toml / uv.lock の手動追随が必須**。

## 事前確認

- ロック対象の `torch 2.12.1+cu126` は cu126 インデックス
  （download.pytorch.org/whl/cu126）に cp314 の Linux x86_64 wheel が存在する
- 他の依存が 3.14 未対応なら `uv lock` の再生成時点で判明する
  （実際は 82 パッケージすべて解決に成功）

## 変更内容

dependabot ブランチに追いコミットして PR #277 をそのままグリーンにする方針。

1. **main への追随**: ブランチが main より古く（torch `>=2.12.0` のまま。main は
   `>=2.12.1`）、そのまま再ロックすると main と競合するため、先に origin/main を
   マージ。force push はリポジトリ運用上避け、origin 先端へのマージ +
   cherry-pick で fast-forward push できる履歴にした
2. **`app/pyproject.toml`**: `requires-python = "==3.11.*"` → `"==3.14.*"`
   （/usr/local のシステム Python を uv にそのまま使わせるため厳密一致で固定）
3. **`app/uv.lock` 再生成**: `cd app && uv lock`。
   torch `2.12.1+cu126` / torchvision `0.27.1+cu126` / transformers `5.12.1` /
   diffusers `0.39.0` が cp314 で解決
4. **`app/Dockerfile`**: dependabot の変更（3.14.6-slim-bookworm）をそのまま採用
5. **`.github/workflows/lint_python.yml`**: ruff 用の `python-version` を
   `"3.11"` → `"3.14"` に追随
6. ルートの `pyproject.toml`（modal クライアント側、`>=3.12`）はビルド失敗と
   無関係のため変更しない

## 検証

- `cd app && docker build . -f Dockerfile` をローカル実行し、CI と同一の
  `uv sync --frozen --no-dev --no-cache` が Python 3.14.6 で完走することを確認
- リンタ: ruff（CI 同条件）/ prettier / tombi パス
- push 後、PR #277 の全チェック（Docker image CI・CodeQL 含む）がグリーン
- `make app_img` での Modal 実機検証（デプロイ → 推論）もユーザーが実施し問題なし

## 運用メモ

- gitignore 済みのローカル `.python-version`（ルート・app/ 両方）は uv の
  インタプリタ解決に影響するため、`uv python pin 3.14` で追随が必要だった
  （コミット対象ではない）
- 今後ベースイメージの Python マイナー/メジャーバンプが来たら、同様に
  `requires-python` の更新と `uv lock` 再生成をセットで行うこと
